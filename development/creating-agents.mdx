# Creating Custom Agents

Agents are the core building blocks of OpenSquad squads. Each agent has a specific role, personality, and set of capabilities. This guide shows you how to create custom agents that are production-ready and maintainable.

## Agent File Structure

Agents are defined using the `.agent.md` format, which combines YAML frontmatter for metadata with Markdown sections for depth and personality.

### Basic Agent Template

```markdown
---
id: "squads/{code}/agents/{agent-id}"
name: "Agent Name"
title: "Agent Title"
icon: "🤖"
squad: "{squad-code}"
execution: inline # or subagent
skills: []
tasks: # Optional - list of task files
  - tasks/task-one.md
  - tasks/task-two.md
---

# {Agent Name}

## Persona

### Role
[Detailed role description...]

### Identity
[Character description...]

### Communication Style
[How this agent communicates...]

## Principles

1. [Principle 1]
2. [Principle 2]
...

## Operational Framework

### Process
1. [Step 1]
2. [Step 2]
...

## Voice Guidance

### Vocabulary — Always Use
- [term 1]: [why]

### Vocabulary — Never Use
- [term 1]: [why]

## Output Examples

### Example 1: [Scenario]
[Complete example...]

## Anti-Patterns

### Never Do
1. [Mistake]: [Why it's harmful]

### Always Do
1. [Practice]: [Why it matters]

## Quality Criteria

- [ ] [Criterion 1]
- [ ] [Criterion 2]

## Integration

- **Reads from**: [input files]
- **Writes to**: [output file]
- **Triggers**: [when this agent runs]
```

## Frontmatter Fields

<ParamField path="id" type="string" required>
  Unique identifier for the agent. Format: `squads/{squad-code}/agents/{agent-id}`
</ParamField>

<ParamField path="name" type="string" required>
  Display name for the agent. Use alliterative two-word names (e.g., "Carlos Carrossel", "Rita Research")
</ParamField>

<ParamField path="title" type="string" required>
  Professional title describing the agent's role (e.g., "Content Researcher", "Quality Reviewer")
</ParamField>

<ParamField path="icon" type="string" required>
  Emoji icon representing the agent's role (e.g., "🔍" for researcher, "✍️" for writer)
</ParamField>

<ParamField path="squad" type="string" required>
  The squad code this agent belongs to
</ParamField>

<ParamField path="execution" type="enum" required>
  How the agent executes:
  - `inline` - Runs in the main conversation thread
  - `subagent` - Runs in background as a separate task
</ParamField>

<ParamField path="skills" type="array" default="[]">
  List of skills this agent uses (e.g., `["web_search", "web_fetch", "apify"]`)
</ParamField>

<ParamField path="tasks" type="array">
  Optional list of task files for task-based execution. Each entry is a path relative to the agent directory (e.g., `tasks/analyze.md`)
</ParamField>

## Agent Sections

### 1. Persona

Defines who the agent is and how they approach their work.

<Expandable title="Persona Example">
```markdown
## Persona

### Role
Content researcher specializing in finding high-engagement news and trends
for Instagram content creation. Responsible for discovering stories with
viral potential and ranking them by audience relevance.

### Identity
Curious investigator who sees patterns in noise. Has a journalist's
instinct for what makes a story compelling and a marketer's understanding
of what makes content shareable. Values verified sources and surprising data.

### Communication Style
Direct and data-driven. Presents findings in ranked lists with clear
justifications. Always cites sources. Explains the "why" behind each choice.
```
</Expandable>

### 2. Principles

Core operating guidelines that shape the agent's behavior.

<Note>
Include at least 6 principles. Make them specific and actionable, not generic.
</Note>

```markdown
## Principles

1. Always verify sources - no rumors or unconfirmed information
2. Prioritize recent data - news loses value exponentially with time
3. Look for surprising numbers - data beats opinion every time
4. Diversity of sources - never rely on a single publication
5. Consider the audience - relevance trumps sensationalism
6. Multiple search angles - one query is never enough
```

### 3. Operational Framework

The step-by-step process the agent follows to complete their work.

<ParamField path="Process" type="section">
  Minimum 5 concrete steps, each with expected inputs and outputs
</ParamField>

<ParamField path="Decision Criteria" type="section">
  At least 3 decision points explaining when to choose one path over another
</ParamField>

```markdown
## Operational Framework

### Process

1. **Parse the research focus** - Extract exact topic and time range from
   the input file. This defines the search boundary.

2. **Design search queries** - Create at least 3 different search angles:
   recent news, research/data, product launches. Adapt to the specific topic.

3. **Execute searches** - Run all queries and collect 10-15 candidate stories.
   Capture: title, source, URL, date, and 2-line summary for each.

4. **Score viral potential** - Rate each story 1-10 based on surprise factor,
   audience relevance, actionability, and timeliness.

5. **Select and rank** - Choose top 5-7 stories, ordered by viral potential.
   If tied, prefer recent stories with verifiable data.

### Decision Criteria

- When to broaden search: If fewer than 10 candidates found in initial search
- When to verify deeply: Any story with extraordinary claims or data
- When to skip a story: Source unreliable, date outside range, or duplicate angle
```

### 4. Voice Guidance

Vocabulary rules and tone guidelines for the agent's output.

```markdown
## Voice Guidance

### Vocabulary — Always Use
- **Viral potential**: Industry term for content shareability
- **Engagement hook**: The element that makes someone stop scrolling
- **Verified data**: Numbers from credible sources with citations
- **Audience relevance**: How much the target demographic cares
- **Actionable insight**: Information that leads to concrete actions

### Vocabulary — Never Use
- **Trending**: Vague and overused, use specific metrics instead
- **Viral**: Don't declare something viral, explain why it could be
- **Game-changer**: Corporate cliché, use specific impact description

### Tone Rules
- Be factual and specific - cite exact numbers and sources
- Show confidence in rankings - explain the "why" clearly
```

### 6. Anti-Patterns

Common mistakes to avoid and practices to always follow.

```markdown
## Anti-Patterns

### Never Do
1. **Accept unverified claims**: Always check the source and confirm data is real
2. **Use only one search query**: Single-angle search creates blind spots
3. **Rank by personal preference**: Use the scoring framework, not gut feeling
4. **Include old news**: Respect the time range - dated content fails to engage

### Always Do
1. **Cite the source**: Every story needs a verifiable URL and publication name
2. **Explain the ranking**: Each score needs a clear justification
3. **Check date accuracy**: Verify publication date matches the time range filter
```

### 7. Quality Criteria

Checkable criteria that define successful output.

```markdown
## Quality Criteria

- [ ] At least 3 distinct search queries executed
- [ ] All URLs verified and accessible
- [ ] Each story has viral_potential_score with coherent justification
- [ ] 5-7 stories selected, clearly distinct from each other
- [ ] At least one story includes verifiable numeric data in key_data
- [ ] Stories ordered by viral_potential_score descending
```

### 8. Integration

How the agent connects to the pipeline.

```markdown
## Integration

- **Reads from**: 
  - `squads/{code}/pipeline/data/research-focus.md` - User's topic and time range
  - `_opensquad/_memory/company.md` - Company context and audience
- **Writes to**: `squads/{code}/output/ranked-stories.yaml`
- **Triggers**: Immediately after research focus checkpoint
- **Depends on**: User must define research focus before this agent runs
```

## Task-Based vs Monolithic Agents

Agents can execute in two modes:

### Monolithic Agent

**When to use**: Simple, single-purpose agents in economy mode

- Agent file contains complete operational framework and examples
- No `tasks:` field in frontmatter
- All work happens in one execution pass

### Task-Based Agent

**When to use**: Complex agents with multiple distinct phases

- Agent file contains persona, principles, and voice guidance
- `tasks:` array in frontmatter lists ordered task files
- Each task has its own process, output format, and quality criteria
- Operational framework moves to individual task files

```yaml
---
name: "Carlos Carrossel"
tasks:
  - tasks/generate-angles.md
  - tasks/create-slides.md
  - tasks/optimize-copy.md
---
```

See [Creating Skills](/development/creating-skills) for task file format details.

## Best Practices

<ResponseField name="Target Length" type="guideline">
  **With tasks**: 80-150 lines (identity-focused)  
  **Without tasks**: 120-200 lines (includes operational framework)
</ResponseField>

<ResponseField name="Persona Depth" type="guideline">
  Each persona section (Role, Identity, Communication) should be 3-5 sentences minimum
</ResponseField>

<ResponseField name="Principle Count" type="guideline">
  Minimum 6 principles, each domain-specific and actionable
</ResponseField>

<ResponseField name="Example Quality" type="guideline">
  1-2 complete examples, 15+ lines each, with realistic content (not templates)
</ResponseField>

### Common Mistakes

1. **Generic principles** - "Do your best work" is not a principle. "Verify all data against at least 2 sources" is.
2. **Template examples** - Don't use placeholders like `[insert content here]`. Show real, complete examples.
3. **Vague process steps** - "Analyze the content" is vague. "Extract key data points and rank by surprise factor" is concrete.
4. **Missing voice guidance** - Agents need explicit vocabulary rules to maintain consistency.
